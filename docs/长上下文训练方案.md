# MindLM 长上下文训练方案

## 概述

将 MindLM 从当前 1K 上下文扩展到 4K、8K、32K，需要综合运用多种显存优化技术。本文档梳理了完整的实现方案，按阶段递进。

---

## MindLM 的天然优势

MindLM-1B 有 16 层，其中 12 层是 Linear Attention（O(T)），只有 4 层是标准 Attention（O(T²)）。这意味着长上下文扩展比纯标准 Attention 模型容易得多。

```
12 层 Linear Attention: 天然 O(T)，不吃显存，不需要任何优化
 4 层标准 Attention:    O(T²)，是显存瓶颈，需要优化
```

---

## 显存优化技术栈

从易到难排列，按需组合：

### 1. Flash Attention（已有，无需改动）

**原理**：不物化完整的 attention 矩阵，用分块计算直接得到结果。

```
标准:  存完整 T×T 矩阵 → 显存 O(T²)
Flash: 不存，分块算    → 显存 O(T)
```

MindLM 已集成（`modeling_mindlm.py` 第 117 行 `scaled_dot_product_attention`），PyTorch 2.0+ 自动走 Flash Attention 路径。

**适用**：所有阶段，是基础。

---

### 2. 梯度检查点（Gradient Checkpointing，一行代码）

**原理**：前向传播时不保存中间激活值，反向传播时重新计算，用时间换空间。

```python
# 在 init_model() 中加一行
model = MindLM(config).to(args.device)
model.gradient_checkpointing_enable()  # 加这一行
```

| 指标 | 不开启 | 开启 |
|------|--------|------|
| 激活显存 | ~50GB (seq=4096, batch=80) | ~20GB |
| 训练速度 | 100% | ~75-80% |

**适用**：4K/8K 阶段，batch_size 较大时。

---

### 3. 降低 batch_size + 梯度累积（改启动参数）

**原理**：显存与 batch_size 近似线性关系，降低 batch_size 直接减显存，用梯度累积保持有效 batch 不变。

```bash
# 效果完全等价
--batch_size 80 --accumulation_steps 2   # 单卡需要 ~50GB
--batch_size 40 --accumulation_steps 4   # 单卡需要 ~25GB
--batch_size 20 --accumulation_steps 8   # 单卡需要 ~13GB
# 有效 batch 都是 80×2 = 40×4 = 20×8 = 160
```

**适用**：任何阶段，最灵活的调节手段。

---

### 4. 滑动窗口 Attention（需改模型代码）

**原理**：限制标准 Attention 每个 token 只看最近 W 个 token，复杂度从 O(T²) 降为 O(T×W)。

```
标准:  每个 token 看所有 token → 矩阵 T×T
窗口:  每个 token 只看最近 W 个 → 矩阵 T×W (W << T)
```

详细设计见 `docs/sliding_window_design.md`。

| 上下文 | 窗口大小 | 是否需要 |
|--------|---------|---------|
| 4K | 不需要 | Flash Attention 够了 |
| 8K | 不需要 | 4 层 8K 显存吃得住 |
| 32K | 4096 | 必须用，否则 4 层 32K 显存爆炸 |

**注意**：只影响 4 层标准 Attention，12 层 Linear Attention 不受影响，仍然保持全局视野。

**适用**：32K 阶段。

---

### 5. RoPE 位置编码扩展（需改模型代码）

**原理**：直接扩大 max_seq_len 会让模型遇到训练时从未见过的位置编码角度，导致注意力崩溃。NTK-aware Scaling 通过调整频率基数 theta 来平滑过渡。

```
原始:   theta = 10000
缩放后: theta_new = theta × scale^(d/(d-2))

scale = new_seq_len / original_seq_len
```

详细设计见 `docs/rope_extension_design.md`。

| 阶段 | max_seq_len | rope_scaling_factor | theta_new |
|------|-------------|-------------------|-----------|
| Stage 1 | 2048 | 1.0 | 10,000 |
| Stage 2 | 8192 | 4.0 | ~40,900 |
| Stage 3 | 32768 | 16.0 | ~167,000 |

**适用**：任何上下文扩展阶段。

---

### 6. FSDP（替换 DDP，多卡共享显存）

**原理**：DDP 每张卡存完整模型，FSDP 把参数/梯度/优化器状态切片分布到多张卡。

```
DDP:  每张卡: 模型 + 优化器 + 梯度 = ~1.2GB × N卡 = 浪费
FSDP: 每张卡: (模型+优化器+梯度)/N = ~0.6GB × N卡 = 省一半
```

代码改动很小，把 DDP 换成 FSDP：

```python
# 原来
from torch.nn.parallel import DistributedDataParallel as DDP
model = DDP(model, device_ids=[rank])

# 改成
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
model = FSDP(model)
```

**适用**：模型参数量更大时（500M+），或者需要省出更多显存给激活值时。

---

### 7. Ring Attention（超长上下文，暂不需要）

**原理**：把序列本身切到多张卡，Q 沿着环形依次跟每张卡的 K/V 算 attention。

```
GPU0: token 0-1023    的 Q K V
GPU1: token 1024-2047 的 Q K V
GPU2: token 2048-3071 的 Q K V
GPU3: token 3072-4095 的 Q K V
→ 序列并行，每张卡只负责 1/4 的序列
```

**适用**：128K+ 超长上下文，MindLM 暂时不需要。

## 分阶段实施方案

### Stage 1：当前预训练（1K 上下文）— 已完成

```
max_seq_len: 1024
技术: DDP（2×A800）
batch_size: 80, accumulation: 2
显存: 充裕
```

---

### Stage 2：4K 上下文扩展

**需要的优化**：梯度检查点 + 降低 batch_size

```bash
torchrun --nproc_per_node=2 pretrain.py \
    --resume_from out/mindlm_pretrain_768_linear_epoch0.pth \
    --model_config mindlm_1b \
    --max_seq_len 4096 \
    --epochs 2 \
    --learning_rate 1e-4 \
    --batch_size 80 \
    --accumulation_steps 2 \
    --ddp \
    --num_workers 8
```

**代码改动**：

1. `pretrain.py`：加 `model.gradient_checkpointing_enable()`
2. `pretrain.py`：加 `--resume_from` 和 `--max_seq_len` 参数（已有）
3. `modeling_mindlm.py`：加 RoPE NTK-aware scaling（推荐但非必须）

**如果 OOM**：`--batch_size 40 --accumulation_steps 4`

---

### Stage 3：8K 上下文扩展

**需要的优化**：梯度检查点 + RoPE NTK + 可能降 batch

```bash
torchrun --nproc_per_node=2 pretrain.py \
    --resume_from out/mindlm_pretrain_768_stage2.pth \
    --model_config mindlm_1b \
    --max_seq_len 8192 \
    --epochs 2 \
    --learning_rate 1e-4 \
    --batch_size 40 \
    --accumulation_steps 4 \
    --ddp \
    --num_workers 8
```

**代码改动**：

1. 所有 Stage 2 的改动
2. `modeling_mindlm.py`：加 `rope_scaling_factor=4.0`（NTK-aware）
3. `config/mindlm_1b.json` 或新建 `config/mindlm_1b_8k.json`

**显存估算**（2×A800, batch=40, seq=8192）：

```
模型+优化器: ~1.2GB
激活值(梯度检查点): ~25GB
总计: ~26GB/卡  ← 没问题
```

---

### Stage 4：32K 上下文扩展（可选）

**需要的优化**：梯度检查点 + RoPE NTK + 滑动窗口 + 降 batch

```bash
torchrun --nproc_per_node=2 pretrain.py \
    --resume_from out/mindlm_pretrain_768_stage3.pth \
    --model_config mindlm_1b_32k \
    --epochs 1 \
    --learning_rate 5e-5 \
    --batch_size 16 \
    --accumulation_steps 8 \
    --ddp \
    --num_workers 8
```

**代码改动**：

1. 所有 Stage 3 的改动
2. `modeling_mindlm.py`：实现滑动窗口 Attention（见 `docs/sliding_window_design.md`）
3. `config/mindlm_1b_32k.json`：`max_seq_len=32768, sliding_window=4096, rope_scaling_factor=16.0`

**显存估算**（2×A800, batch=16, seq=32768, 滑动窗口4K）：

```
模型+优化器: ~1.2GB
激活值(梯度检查点): ~20GB
滑动窗口attention: ~2MB（块状实现）
总计: ~22GB/卡  ← 没问题
```

## 推荐实施顺序

```
现在: 跑 SFT（当前 1K 上下文）
  ↓
第一步: 4K 扩展（梯度检查点 + 可能降 batch）
  ↓      改动量：~5 行代码
  ↓
第二步: 8K 扩展（+ RoPE NTK）
  ↓      改动量：~20 行代码
  ↓
第三步: 评估效果，决定是否上 32K
  ↓      改动量：~50 行代码（滑动窗口）
```

每一步都只在前一步跑通、效果验证后再推进，不需要一次性全做。
